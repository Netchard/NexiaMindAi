# Accessibility Review — NexiaMind AI Accueil

## Overall verdict

Not AA-ready as written. The spec gets the *intent* right in several places (aria-disabled kept focusable, hover+focus tooltip trigger, visible focus ring, semantic buttons) but has one confirmed hard contrast failure affecting both primary conversion CTAs, a second confirmed failure on a load-bearing label, and several behavioral gaps (tooltip persistence, SPA route-change announcement, aria-disabled's non-native activation-blocking) that are exactly the kind of thing that ships broken if left to per-developer judgment. None of these require a redesign — all are fixable with token/behavior amendments — but they should be resolved in DESIGN.md/EXPERIENCE.md before hand-off to implementation, not discovered in QA.

### Findings

- **critical** `{colors.on-primary}` (`#FFFFFF`) on the `cta-primary`/`closing-cta` gradient (`{colors.hero-cta-gradient-from}` `#F4693F` → `{colors.hero-cta-gradient-to}` `#E64F2B`) fails WCAG 1.4.3. Computed contrast is ~3.03:1 at the `#F4693F` end and ~3.81:1 at the `#E64F2B` end — both below the 4.5:1 required for normal text. `cta-label` is 15.5px/600, which does not qualify as "large text" under either reading of the AA large-text exception (needs ≥24px regular, or ≥18.66px/14pt at font-weight ≥700 — 15.5px clears neither, and 600 is semi-bold, not bold). This pair is used on the two primary conversion actions of the page (hero CTA + closing CTA) and DESIGN.md never states a contrast value for it (DESIGN.md §Colors, §Components `cta-primary`/`closing-cta`; also present verbatim, unflagged, in the inherited chat DESIGN.md). *Fix:* either darken the gradient stops (a single darker/more saturated orange, e.g. closer to `#C14E2F`-class values, checked against white at 4.5:1) or specify a heavier/larger `cta-label` that legitimately clears the large-text 3:1 bar (e.g. ≥19px at true bold 700) — and record the resulting contrast ratio in DESIGN.md so it isn't re-broken on the next token edit.

- **high** `{colors.chips-eyebrow-text}` (`#6F7789`) on `{colors.surface}` (`#0A1524`) computes to ~4.08:1, below the 4.5:1 threshold for normal text. `chips-eyebrow` is 13px/600 — far under the large-text size floor regardless of weight, so the exception doesn't apply. This is the "Essayez une question — cliquez pour lancer le chat" instructional label directly above the chips — not decorative, and arguably load-bearing for users who don't already know clicking a chip immediately starts a conversation (DESIGN.md §Colors, §Typography `chips-eyebrow`). *Fix:* lighten the token (something in the `#8D9CB5`/`ink-subtle`-to-`ink-muted` range clears 4.5:1 against this surface with margin) and re-verify.

- **medium** `cta-secondary-disabled` text contrast is unverified and implementation-dependent. In isolation, `{colors.cta-secondary-text}` (`#DFE4EE`) on `{colors.cta-secondary-bg}` computes to a comfortable ~13.4:1, but DESIGN.md applies `disabledOpacity: 0.55` to the whole component ("rendu à 0.55... et `cursor: not-allowed`"), which — if implemented as a single CSS `opacity` on the button (the natural reading, and the common Tailwind `opacity-55` pattern) — composites text, border and background against the page surface behind it. That drops effective text contrast to roughly ~5:1, uncomfortably close to the 4.5:1 floor and sensitive to whatever's actually rendered behind the button (DESIGN.md never states the *resulting* ratio, only the input opacity). Whether this control even qualifies for WCAG's "inactive UI component" contrast exemption is itself debatable: the exemption assumes a genuinely inert control, but EXPERIENCE.md deliberately keeps this one focusable, `aria-describedby`-labeled and exposed to the accessibility tree specifically so the tooltip is discoverable — i.e. it's designed to still read as "an interactive thing" to AT, which cuts against treating it as exempt (DESIGN.md §Components `cta-secondary-disabled`; EXPERIENCE.md §Accessibility Floor). *Fix:* don't dim via a blanket element-level `opacity`; specify the disabled-state text/border/background as discrete solid tokens, compute and record their contrast against `{colors.surface}` directly, and target ≥4.5:1 for the label text regardless of the exemption question.

- **high** `aria-disabled="true"` alone does not prevent activation — it needs to be paired with logic the spec never calls out explicitly. Unlike the native `disabled` attribute, `aria-disabled` is purely a semantic/announcement hint; the element remains clickable and, if it's a `<button>`, still fires on Enter/Space unless the click/keydown handler is manually gated. EXPERIENCE.md leaves the element choice open ("`<button>`/`<a>` avec `role` approprié") and separately states "pas de `href` fonctionnel" for the secondary CTA, which reads as license to implement it as an `<a>` with no (or a dummy) `href` — but an anchor without `href` is not a link in the accessibility tree and is **not** natively focusable, contradicting the same section's requirement that it "reste focusable... pour que la tooltip soit atteignable au clavier." Getting this right requires a `<button type="button">` (focusable, Enter/Space handled by the platform, only needs the click handler gated) or an `<a>` retrofitted with `tabindex="0"`, `role="button"` and manual keydown handling — none of which the spec mandates (EXPERIENCE.md §Component Patterns, §Accessibility Floor). *Fix:* mandate `<button type="button" aria-disabled="true">` explicitly (removes the anchor ambiguity entirely) and add one line stating the click/keydown handler must still be gated in code — `aria-disabled` is cosmetic to the browser, not enforcement.

- **medium** Tooltip on the disabled CTA is only specified as appearing on hover/focus — WCAG 1.4.13's other two requirements (dismissible via Esc, persistent until dismissed/blur, not disappearing before the pointer can reach it) are never addressed. EXPERIENCE.md's only statement is the single State Patterns row: "Survol / focus → Tooltip 'Bientôt disponible' apparaît" (EXPERIENCE.md §State Patterns, §Accessibility Floor). Since the same information is already exposed programmatically via `aria-describedby` (so screen reader users get it without the visual tooltip ever needing to render), the residual risk is for sighted keyboard users: nothing says the tooltip can be dismissed with Escape without moving focus off the button, or that it won't auto-hide on a timer. *Fix:* add a line specifying Esc dismisses the tooltip without blurring the button, and that it remains visible for as long as the button has hover or focus (no timeout).

- **medium** Whether the tooltip's text node exists in the DOM before it's triggered is unspecified, which matters for `aria-describedby`. If the tooltip content is only mounted on hover/focus (common implementation shortcut) rather than always present and merely visually hidden, the `id` that `aria-describedby` references may not exist yet at the moment focus lands, and screen reader support for `aria-describedby` targets that appear *after* focus is inconsistent across AT/browser combinations (EXPERIENCE.md §Accessibility Floor references `aria-describedby` but not DOM-mount timing). *Fix:* state explicitly that the tooltip text element is always present in the DOM (`visibility:hidden`/`opacity:0` when inactive, not conditionally rendered), so the `aria-describedby` reference is valid from first render.

- **medium** Auth-guard latency at click time is not addressed. EXPERIENCE.md asserts the redirect for unauthenticated users is immediate with no loading state needed ("aucun état de chargement propre à l'accueil... la redirection est immédiate," §Foundation, §State Patterns) but this assumes `useAuth()` already has a resolved, synchronous answer at the moment of click. If auth status is still being determined (e.g. a session/token check in flight shortly after first paint — `/` is public and un-gated at the route level specifically so it renders before auth resolves), a click landing in that window has no specified behavior: neither "acts as logged in," "acts as logged out," nor "waits" is chosen. A keyboard/AT user who tabs to the CTA and activates it quickly after page load is the likeliest to hit this window. *Fix:* specify the click behavior while auth status is indeterminate (e.g. treat as not-authenticated by default, or briefly disable actionable controls until the first auth resolution) so a click can't silently no-op.

- **high** No focus/announcement handling is specified for the abrupt client-side navigation triggered by chips and CTAs. Clicking a chip calls `onSendMessage` and navigates to `/chat` via client-side routing the moment the conversation is created (EXPERIENCE.md §State Patterns, "la navigation vers `/chat` a lieu dès que la conversation est créée"). In a Next.js App Router SPA transition, this does **not** get the automatic screen-reader "new page" announcement a full page load would (no title-change/focus-reset-to-body event fires the way it does on hard navigation) — this is a well-known SPA accessibility gap, generally addressed via WCAG 4.1.3-style status-message handling or explicit focus management on route change. Nothing in EXPERIENCE.md's Component Patterns, State Patterns, or Key Flows sections moves focus to the new page's heading/content or announces the transition; a screen reader user is left exactly where the review brief worried about — silently dropped into a new page context with focus wherever it happened to be. This affects every successful chip/CTA click, not an edge case. *Fix:* specify that on arrival at `/chat` from an accueil action, focus moves to a sensible landmark (e.g. the chat's main heading or the newly-created message) — likely a note that belongs in the chat EXPERIENCE.md's route-entry state, cross-referenced from here.

- **medium** Heading semantics are unspecified. Neither DESIGN.md nor EXPERIENCE.md states that the hero title ("Toute la connaissance de l'entreprise...") is an `<h1>`, that "Trois pas, et vous avez la réponse" is an `<h2>`, or what level the step-card titles ("Demandez"/"Nexia cherche"/"Réponse sourcée") use. Both are described purely in visual/typographic terms (`{typography.hero-display}`, `{typography.section-display}`, `{typography.step-title}` — DESIGN.md §Components) with no semantic tag called out, and EXPERIENCE.md doesn't fill the gap either. Given the page also lives under a shared app-shell `Navbar` (potentially with its own heading/brand element), there's a real risk of the hero title being implemented as a styled `<div>`/`<p>` rather than the page's `<h1>`, or of a duplicate/missing `<h1>`. *Fix:* add one line to DESIGN.md or EXPERIENCE.md stating hero-heading is `<h1>`, section-display is `<h2>`, step-title is `<h3>`.

- **low** The badge's pulsing dot has real motion behind it that neither spec file defines, and `prefers-reduced-motion` is unaddressed anywhere in the document. The imported maquette (`imports/maquette-nexiamind-ai-accueil-2026-07-18.html`) defines an actual `@keyframes pulseGlow { 0%,100% { opacity: 0.55; } 50% { opacity: 0.9; } ... }` continuous loop, and EXPERIENCE.md's own accessibility line calls it "le point pulsant" (§Accessibility Floor) — confirming the pulse is intended to carry into implementation — yet DESIGN.md only records a static `box-shadow: 0 0 10px {colors.accent-orange}` halo, dropping the animation definition (duration/easing) entirely and never mentioning reduced-motion. Note for calibration: a slow, non-flashing opacity pulse like this isn't a strict WCAG 2.2 AA failure on its own (2.3.3 Animation from Interactions is AAA and only covers interaction-triggered motion; 2.3.1's flash threshold is nowhere near this), so this is a best-practice gap rather than a confirmed violation — but it's an easy, cheap fix and the kind of thing worth closing before it's someone's forgotten follow-up. The CTA hover `filter: brightness(1.08)` and chip hover transitions are lower-risk (subtle, non-looping, user-triggered) but would also benefit from being covered by the same blanket rule. *Fix:* wrap the pulse animation in `@media (prefers-reduced-motion: reduce)` → static halo, no keyframe loop; note this once as a page-wide rule rather than per-element.

- **low** The generic keyboard rule ("Entrée/Espace sur un élément focusé déclenche la même action qu'un clic," EXPERIENCE.md §Interaction Primitives) is stated without an explicit carve-out for the disabled secondary CTA, which should *not* activate on Enter/Space per its own "aucune action au clic" rule a few lines earlier (§Component Patterns). The two rules aren't actually contradictory once read together, but a spec that has to be cross-referenced to resolve an apparent conflict is a spec that risks being implemented wrong. *Fix:* add a one-clause exception to the keyboard rule ("...à l'exception du CTA secondaire désactivé, où Entrée/Espace n'a aucun effet, comme le clic").

### Not flagged (checked, passes with margin)

- `hero-heading` (#F6F8FC), `hero-subheading` (#A8B0C0), `chip-text` (#CDD4E0 at rest and white-on-hover), `step-body-text` (#98A1B3), and `accent-orange` (#FF8A56) in its three text roles (eyebrow badge text, step-number-badge digit, hero italic word) all clear their respective AA thresholds against their stated backgrounds, with margins ranging from ~4.1:1 of headroom (hero-heading, effectively ~17:1) down to a tighter but still-passing ~6.5:1 (step-body-text against the lightest point of the step-card gradient).
- The stated Tab order (hero non-focusables → primary CTA → disabled CTA → chips → step cards non-focusable → closing CTA) matches the described top-to-bottom visual/DOM layout and doesn't appear to require any CSS reordering that would decouple them — no WCAG 2.4.3 issue found here.
